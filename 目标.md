# 当前架构背景

我的系统已经改成了“按视频源共享 Session”的架构：

- 一个视频源对应一个 `SourceSession`

- ```
  SourceSession
  ```

   内部维护：

  - `stream`
  - `reader_thread`
  - `infer_thread`
  - `latest_frame`
  - `latest_result`
  - `subscribers`
  - `ref_count`

- 后台定时检测任务和前端 WebSocket 都会复用同一个 `SourceSession`

- 当前问题是：

  - 即使只有后台配置任务，没有任何前端用户，单路 CPU 也很高
  - 共享 Session 后，单路 CPU 升到了 50~60%
  - 之前是三路视频才接近这个占用

# 我希望你重点检查和修改的点

## 1. 把“检测结果”和“前端展示结果”分离

请把 Session 内部结果拆成至少两层：

### A. 检测结果（后台逻辑必须保留）

用于：

- 事件过滤
- 事件判定
- 截图
- 落库
- 告警
- 业务日志

例如：

- bbox
- confidence
- track_id
- event_state
- screenshot_path
- detection_timestamp

### B. 展示结果（只有前端订阅时才生成）

用于：

- 画框
- resize
- JPEG 编码
- WebSocket 推送

要求：

- 如果 `ws_subscribers_count == 0`，则不要生成展示结果
- 如果 `ws_subscribers_count > 0`，才允许生成展示图和 JPEG bytes

------

## 2. 检查并修复忙轮询

请重点检查以下循环是否存在空转：

- `reader_loop`
- `infer_loop`
- `ws_endpoint` 或订阅发送循环
- 后台 detection manager 的监控循环
- reconnect loop
- cleanup loop

要求：

- 避免 `while True` 中无阻塞、无 sleep、高频空转
- 合理使用：
  - `time.sleep(...)`
  - `asyncio.sleep(...)`
  - `queue.get(timeout=...)`
  - `threading.Event`
  - 条件变量
- 不允许在没有新帧/新结果时无限循环占满 CPU

------

## 3. 引入 frame_version / result_version 机制

我怀疑当前实现里即使没有新帧或没有新推理结果，也在重复处理和重复发送。

请你实现：

### frame_version

- Reader 每读到一帧新帧时递增
- Infer 只在 `frame_version` 变化时才处理

### result_version

- Infer 每生成一份新结果时递增
- WebSocket 只在 `result_version` 变化时才发送
- 没有新结果时不要重复 JPEG 编码，不要重复发送同一帧

------

## 4. 避免重复 JPEG 编码

请检查当前逻辑是不是每个订阅者都在重复做：

- draw box
- cv2.imencode('.jpg', frame)

要求改成：

- infer 或 session 内部统一生成一次展示帧 / 一次 JPEG bytes
- 多个订阅者共享同一份已编码结果
- 不允许每个 WebSocket 各自重复编码同一帧

------

## 5. 后台模式优化

如果当前 Session 只有后台任务持有，没有任何前端 WebSocket 订阅者：

- 仍然正常读流、推理、事件过滤
- 但不要做前端展示链路的任何操作
- 不要生成无意义的 `latest_encoded_bytes`
- 不要执行无意义的 broadcast/send 逻辑

也就是说，请把 Session 变成两种运行模式：

### 模式 A：后台检测模式

- 有 holder
- 无 ws subscriber
- 只做检测和业务逻辑

### 模式 B：检测 + 实时订阅模式

- 有 holder 或无 holder
- 有 ws subscriber
- 在检测基础上再增加画框、编码和推送

我希望你重点排查并修复这些高概率问题：
