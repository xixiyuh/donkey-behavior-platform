完整业务逻辑流程
1. 视频源获取与处理
1. 视频源输入 ：
  - 用户通过 WebSocket 连接指定视频源类型（ kind ）和路径/URL（ value ）
  - 支持多种视频源：
    - camera ：USB 摄像头
    - file ：本地视频文件
    - rtsp ：RTSP 流
    - image ：静态图片
    - mpv ：HLS/RTMP/HTTP-FLV 等流（使用 MPV 解码）
2. 视频流解码 ：
  - 对于 HLS/RTMP/HTTP-FLV 等流：使用 MPVPipeStream 类
    - 调用 MPV 命令行工具进行解码
    - 通过 stdout 读取 rawvideo 格式数据
    - 转换为 OpenCV 可处理的 BGR 格式
  - 对于其他流：使用 CameraStream 、 FileStream 、 RTSPStream 等类
    - 直接使用 OpenCV 的 VideoCapture 读取
2. 双线程流水线处理
1. Reader 线程 ：
  - 持续从视频源读取帧
  - 低延迟策略：清空旧帧队列，只保留最新帧
  - 实时监控读取帧率
2. Infer 线程 ：
  - 从帧队列获取最新帧
  - 根据 FRAME_INTERVAL 配置决定是否进行推理
  - 调用检测器进行目标检测
  - 在帧上绘制检测框和 FPS 信息
  - 将处理后的帧放入结果队列
3. 模型推理
1. 模型选择 ：
  - Windows 开发环境：使用 PTDetector （PyTorch YOLO 模型）
  - RK3588 部署环境：使用 RKNNDetector （RKNN Lite 模型）
2. 推理流程 ：
  - 输入预处理（letterbox 缩放、格式转换）
  - 模型推理
  - 后处理（NMS、坐标还原到原图）
  - 绘制检测框和标签
4. WebSocket 推送
1. 帧率控制 ：
  - 通过 WSManager 控制最大推送帧率（ MAX_FPS ）
  - 动态调整推送间隔，避免带宽占用过高
2. 数据编码与传输 ：
  - 将处理后的帧编码为 JPEG 格式
  - 进行 Base64 编码
  - 通过 WebSocket 推送给前端
5. 前端显示
1. 接收数据 ：
  - 前端通过 WebSocket 接收 Base64 编码的 JPEG 数据
2. 显示处理 ：
  - 将 Base64 数据转换为图片
  - 在页面上实时显示带检测框的视频流
核心技术点与注意事项
1. 视频流处理
- MPV 解码器 ：
  - 注意 ：必须正确安装 MPV 并配置路径
  - 注意 ：stderr 必须异步读取，否则会导致管道阻塞
  - 注意 ：滤镜链顺序很重要： scale → pad → format
  - 注意 ：固定输出尺寸，否则无法按字节流切帧
- 低延迟策略 ：
  - 注意 ：队列大小设置（ QUEUE_MAX ）影响延迟和稳定性
  - 注意 ：丢帧策略可能导致部分帧被跳过
2. 模型推理
- 模型选择 ：
  - 注意 ：Windows 环境使用 .pt 模型，RK3588 环境使用 .rknn 模型
  - 注意 ：RK3588 对 640x640 分辨率优化最好
- 推理性能 ：
  - 注意 ： FRAME_INTERVAL 设为 2-3 可提升性能，但会降低检测频率
  - 注意 ：RK3588 可配置使用多核心 NPU 提高性能
3. WebSocket 通信
- 带宽优化 ：
  - 注意 ： JPEG_QUALITY 设置影响画质和带宽
  - 注意 ： MAX_FPS 限制推送频率，避免网络拥塞
- 连接稳定性 ：
  - 注意 ：WebSocket 可能因网络问题断开，需要实现重连机制
  - 注意 ：可添加心跳保活机制防止断连
4. 部署与配置
- 环境配置 ：
  - 注意 ：Windows 环境需要安装 MPV 和 PyTorch 依赖
  - 注意 ：RK3588 环境需要安装 RKNN Lite2 依赖
- 性能调优 ：
  - 延迟优化 ：减小队列大小，每帧都推理，提高推送帧率
  - 吞吐量优化 ：增大队列大小，减少推理频率，降低推送帧率
  - 带宽优化 ：降低 JPEG 质量，减小推理分辨率
5. 常见问题与解决方案
- MPV 启动失败 ：检查 MPV 安装和路径配置
- HEVC 解析错误 ：添加更宽松的解析参数
- 帧率不稳定 ：检查 Reader 线程稳定性、推理时间和网络带宽
- WebSocket 断连 ：实现心跳保活机制
- 内存泄漏 ：确保队列正确清理，避免资源未释放
典型使用场景
1. 实时监控 ：通过 RTSP 或 HLS 流监控摄像头画面，实时检测目标
2. 视频分析 ：对本地视频文件进行批量检测分析
3. 边缘计算 ：在 RK3588 开发板上部署，实现低延迟本地检测
4. 远程检测 ：通过网络推流，在服务器端进行检测并返回结果
该系统设计合理，支持多种视频源和部署环境，通过双线程流水线和低延迟策略实现了高效的实时检测功能。
